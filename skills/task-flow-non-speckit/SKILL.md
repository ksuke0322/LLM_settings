# Task Flow (Non-speckit)

speckitを利用しない作業タスクは、`specs/flow/` をデファクトの置き場として運用する。

## コア方針

- **タスク分割**: AIが適切に分割判断する。
  - 原則は「単一目的・単一成果物・限定的な影響範囲」。
  - 目的が複数ある場合や、フロント/バックなど複数領域にまたがる場合は分割を前提にする。
- **ステータス**: `未着手` / `done` のみ。
- **done定義**: そのタスクで定義した期待値がすべて完了している状態。
- **PR要件**: PRの必須項目は別skillの規定に準拠。

## specs/flow ファイルの最低限構成

- タイトル
- 目的
- 方針（必要なら）
- タスク一覧（1タスク=1PR）
- 各タスクの「期待値 / 対象 / 作業」

## 実運用のルール

- 新規タスクは `specs/flow/` に作成する。
- タスク一覧のチェックは `未着手` / `done` のみで管理する。
- done化の判断は「期待値の全達成」が基準。抜けがある場合は done にしない。
- 既存のタスク分割が過大/過小な場合、`specs/flow/` で分割提案を先に行う。
