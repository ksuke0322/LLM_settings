# 開発ワークフロー

- 原則
  - TDD を基本とし、作業前に計画を提示して承認を得る
  - 代替案や方針変更が必要な場合も、必ず承認を得てから実行する

- 手順
  1. 計画提示
     - AI運用5原則の第1原則に則り、要求の要約/実装方針/具体タスク（対象ファイル・正常系/異常系）を提示
     - y/n で承認を取得
  2. TDD ループ
     - テスト作成 → 失敗確認 → 最小実装 → 再実行 → 通過まで反復
     - 原則としてテストは変更しない。妥当な理由がある場合は事前に承認を得る
     - 最初の Red は「仕様として期待する振る舞い」を表現し、その失敗理由を一文で説明する
       - 例: 「未実装のため 401 を返せない」など
     - Green 後に必ずリファクタリングを行い、重複排除・命名・構造を整える（挙動は変えない）
  3. 検証（動作確認コマンドの順序）
     - レポジトリごとに適切な動作確認コマンドを実行（例 : `pnpm lint, pnpm test, pnpm build`）
       - ただしプロダクトコードに関わらないファイルの変更の場合は不要（例 : `README.md`, `docs/`）
     - 失敗時は原因を特定・修正し、同順序で再実行する
  4. 反映
     - 反映前に git diff と git status を確認し、意図しない変更がないことを保証する
     - 変更点の要約、影響範囲、既知の制約、次アクション（提案）を提示
     - PR 作成時は `~/.codex/instructions/git/commit_pr.md` のルールを遵守する
